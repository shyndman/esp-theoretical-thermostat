# Phase 2.1: memory usage reduction, as discussed - Context

**Gathered:** 2026-02-10
**Status:** Ready for planning

<domain>
## Phase Boundary

Urgently reduce runtime RAM pressure and fragmentation discovered during execution, using config-gated tuning changes that remain default-off during this phase, while preserving the required runtime contract and not degrading display quality.

</domain>

<decisions>
## Implementation Decisions

### Reduction Priority and Wave Strategy
- First reduction wave targets WebRTC pools.
- Second priority is MQTT queue depth reduction.
- Remaining work follows largest RAM win next.
- Waves should be large/parallel where possible, not single-change increments.
- Stack tuning policy is: measure high-water marks first, then set stack to high-water plus margin.
- If a large bundle has mixed outcomes, resolution is case-by-case with required human verification.
- Pause further reductions when regressions are dramatic in required runtime behavior, UX/responsiveness, or hard-failure behavior.
- Preserve display quality; do not trade visual quality for RAM wins.
- Do not intentionally degrade performance; remove non-essential overhead instead.

### Validation Matrix
- Mandatory validation scenarios for this phase are boot + idle + touchscreen interaction.
- Acceptance signal is human sign-off only.
- All new tuning flags stay default-off throughout Phase 2.1.
- Failed bundles require failure notes only for triage.

### Required Runtime Contract
- Required capabilities for an acceptable build are network, MQTT, display, and touchscreen.
- If a tuning bundle breaks any required capability at boot, retry once and reject on repeat.
- Non-required features may degrade temporarily during this phase.
- Operator signaling for degraded non-required features is case-by-case.

### Claude's Discretion
- No broad "you decide" areas were delegated for this phase.
- Any mixed-bundle outcomes and degraded-feature signaling remain human case-by-case decisions.

</decisions>

<specifics>
## Specific Ideas

- "Cut the fat" above what runtime truly requires, instead of removing required capabilities.
- Keep display quality intact while reducing memory pressure.
- Use config gating to stage and evaluate tuning changes safely.

</specifics>

<deferred>
## Deferred Ideas

- Continue/complete Phase 3 watchdog strictness and post-fault evidence decision-making after Phase 2.1 RAM stabilization.
- Any new capability expansion beyond RAM attribution/reduction remains out of scope for Phase 2.1.

</deferred>
