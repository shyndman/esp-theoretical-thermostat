## Summary

This firmware is already instrumented enough to run aggressive RAM-reduction waves safely: `runtime_health` exposes stack headroom for MQTT/env/WebRTC/radar tasks, and periodic heap metrics include free/min/largest internal block and risk transitions. That means Phase 2.1 can plan around measured headroom and fragmentation, not guesses.

The largest immediate reduction opportunity is WebRTC peer buffering configuration in `main/streaming/webrtc_stream.c`, where configured pool/cache sizes are very large (`send_pool_size`, jitter caches, and data channel caches). The second high-impact target is MQTT dataplane queue depth in `main/connectivity/mqtt_dataplane.c`; each queue slot carries a full 1 KB fragment buffer, so depth reductions produce deterministic RAM savings. Stack right-sizing should be done in the same wave style the user requested, but only after collecting high-water marks under required scenarios.

Use config-gated bundles (default off) and evaluate in large waves: WebRTC pool bundle first, MQTT queue-depth bundle second, then stack right-sizing and secondary queues/tasks by measured payoff. Keep display/touch quality unchanged; optimize overhead and buffer sizing, not UI fidelity.

**Primary recommendation:** Plan three config-gated bundles in order (WebRTC pools -> MQTT queue depth -> measured stack/secondary buffers), each validated by boot + idle + touchscreen with required contract checks and human sign-off.

## Architecture Patterns

### Recommended Project Structure
```
main/
├── streaming/             # WebRTC pool and capture/peer tuning bundle
├── connectivity/          # MQTT queue depth and runtime health instrumentation
└── sensors/               # Secondary stack right-sizing candidates
```

### Pattern 1: Config-Gated Reduction Bundles (Default Off)
**What:** Group multiple related RAM reductions behind one Kconfig flag per wave; keep every new flag default `n` during Phase 2.1.
**When to use:** Every reduction wave in this phase (WebRTC first, MQTT second, then next-largest win).
**Example:**
```c
// Source: project pattern in main/Kconfig.projbuild + sdkconfig.defaults
// New gate examples for planning (names illustrative):
config THEO_RAM_WAVE1_WEBRTC_TUNING
    bool "Enable RAM wave 1: WebRTC pool tuning"
    default n

config THEO_RAM_WAVE2_MQTT_QUEUE_TUNING
    bool "Enable RAM wave 2: MQTT dataplane queue depth tuning"
    default n
```

### Pattern 2: Measure-Then-Set Stack Policy
**What:** Use `uxTaskGetStackHighWaterMark2()` telemetry to set stack size to high-water plus margin, not by guess.
**When to use:** Any task stack change (MQTT, env sensors, WebRTC worker, diagnostics, radar helper).
**Example:**
```c
// Source: ESP-IDF FreeRTOS docs + existing runtime_health pattern
configSTACK_DEPTH_TYPE hwm = uxTaskGetStackHighWaterMark2(task_handle);
// In ESP-IDF, task stack parameters are specified in bytes.
// Plan: new_stack = observed_peak_used + safety_margin
```

### Pattern 3: Capability-Aware Placement for Non-Critical Buffers
**What:** Move selected buffers/queues to SPIRAM or explicitly choose memory caps for queue/task storage.
**When to use:** After required internal-RAM users are identified and protected.
**Example:**
```c
// Source: ESP-IDF freertos_additions + mem_alloc docs
QueueHandle_t q = xQueueCreateWithCaps(depth, item_size, MALLOC_CAP_SPIRAM | MALLOC_CAP_8BIT);
// Delete with vQueueDeleteWithCaps(q)
```

### Anti-Patterns to Avoid
- **Un-gated tuning changes:** violates the locked decision that all new tuning flags remain default-off in Phase 2.1.
- **Single-change micro-waves:** conflicts with required large/parallel wave execution and slows attribution.
- **Assuming stack units incorrectly:** in ESP-IDF APIs stack sizes are bytes; mixing word assumptions causes bad sizing.
- **Quality trade for RAM:** reducing display buffers/settings that hurt visual quality is explicitly disallowed.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Per-task heap attribution | custom alloc wrappers everywhere | `heap_caps_get_per_task_info` / task-stat APIs | Official APIs already provide per-task totals and block info |
| Allocation timeline tracing | custom malloc/free hooks framework | `heap_trace_*` standalone tracing | Supported tracing modes, summaries, and tooling are already available |
| Queue memory placement hacks | custom queue structs + alloc/free lifecycle | `xQueueCreateWithCaps` + `vQueueDeleteWithCaps` | Safer lifecycle and capability-aware allocation already implemented by IDF |
| WebRTC buffer subsystem | custom RTP/data buffering layer | `esp_peer_default_cfg_t` tuning knobs | Existing stack exposes direct cache/pool/queue controls |

**Key insight:** In this domain, the fastest safe wins come from tuning supported configuration surfaces, not replacing infrastructure.

## Common Pitfalls

### Pitfall 1: WebRTC Pool Tuning Without Directionality Verification
**What goes wrong:** Teams shrink recv jitter/data channel caches aggressively even when unsure what gets allocated for send-only mode.
**Why it happens:** `esp_peer_default_cfg_t` has many knobs and allocation behavior can depend on negotiated media/data settings.
**How to avoid:** First wave should tune obvious oversized fields, but keep uncertain fields behind a separate sub-gate and validate memory + runtime behavior explicitly.
**Warning signs:** Boot-time WebRTC failures, session negotiation failures, or sharp reconnect churn.

### Pitfall 2: MQTT Queue Depth Reduction Without Burst Evidence
**What goes wrong:** Queue depth cut too far causes `DP_DROP_QUEUE_FULL` spikes and delayed/partial state updates.
**Why it happens:** Dataplane queue items are large and copied by value; depth is tempting to cut quickly.
**How to avoid:** Stage depth reductions with digest/drop counter observation and required-contract checks before accepting.
**Warning signs:** Repeated `mqtt_drop reason=queue_full` and missing initial-state readiness.

### Pitfall 3: Stack Right-Sizing by Guess
**What goes wrong:** Stacks become unstable under real runtime paths despite seeming idle-safe.
**Why it happens:** Sizing from intuition instead of high-water marks and margin.
**How to avoid:** Use observed high-water metrics under boot + idle + touchscreen, then apply policy: high-water plus margin.
**Warning signs:** runtime-health WARN/CRIT transitions after stack cuts or intermittent hard faults.

### Pitfall 4: Fragmentation Interpretation Errors
**What goes wrong:** Free heap looks healthy but allocations fail because largest contiguous block collapses.
**Why it happens:** Monitoring only free bytes, not largest block and ratio.
**How to avoid:** Gate decisions on free + minimum + largest-block trend together.
**Warning signs:** decreasing `largest_free_block` while total free bytes stay relatively flat.

### Detect Fragmentation Risk with Largest-Free-Block Ratio
```c
// Source: https://docs.espressif.com/projects/esp-idf/en/release-v5.5/esp32s3/api-reference/system/heap_debug
size_t free_b = heap_caps_get_free_size(MALLOC_CAP_INTERNAL);
size_t largest_b = heap_caps_get_largest_free_block(MALLOC_CAP_INTERNAL);
uint16_t ratio_permille = (free_b == 0) ? 0 : (uint16_t)((largest_b * 1000U) / free_b);
```

## Open Questions

1. **Which `esp_peer_default_cfg_t` fields are allocated in this firmware's send-only profile?**
   - What we know: tuning fields exist and are large; component README documents memory/performance trade-offs.
   - What's unclear: exact runtime allocation impact per field under send-only + data-channel-disabled use.
   - Recommendation: split WebRTC wave into sub-gates and collect before/after internal-heap deltas plus session stability.

2. **How low can MQTT dataplane queue depth go without violating initial-state behavior?**
   - What we know: queue slots carry 1 KB payload buffers and queue-full drops are already instrumented.
   - What's unclear: minimum safe depth during real burst patterns in this deployment.
   - Recommendation: run depth ladder experiments under mandatory validation matrix and reject any repeated required-contract failures.

3. **What stack margins are acceptable per task after high-water measurement?**
   - What we know: policy is high-water plus margin, and probes already exist for critical tasks.
   - What's unclear: margin values per task that remain stable across long runtime.
   - Recommendation: set initial conservative margins, then tighten only after repeated human-verified runs.
